# 上下文契约

每次 subagent 请求都使用以下字段，内容只保留完成该职责所需的最小事实：

```text
目标：要解决的单一问题与完成标准。
约束：允许的写入范围、禁止事项、能力上限和时间/风险限制。
必要文件：精确路径，以及确有必要的已脱敏片段或行号。
已有证据：已运行命令、失败信息、已知行为和未确认假设。
审批证据：approved_design 或 approval_evidence；若属例外，注明纯字面或已批准规格内机械改动。
期望输出：本角色需要给出的结论、建议或补丁范围。
```

## 脱敏规则

在发送任何上下文前，移除密钥、token、个人数据、生产数据、内部 URL 和无关路径。必要值以 `[REDACTED_<TYPE>]` 形式保留类型而不保留原值，例如 `[REDACTED_TOKEN]`、`[REDACTED_INTERNAL_URL]`。禁止附加完整聊天记录和完整日志；长输出先提炼为与目标直接相关的事实、错误片段和命令结果。

## 写入所有权

每个可写 agent 请求必须声明精确文件集合。并行写入的集合必须互斥；若存在任一重叠，主 agent 必须串行协调这些 agent。子 agent 不得再委派其他 agent。

subagent 只按下列结构返回：

```text
结论：直接回答目标。
文件引用：涉及的路径和必要行号。
验证：运行过的命令、结果，或未运行的原因。
剩余风险：未覆盖项、假设和建议升级条件。
```

主 agent 必须独立检查返回的文件引用和验证。
