# 工作原理

Model Economy 是一套工作流策略，不会自动宣称模型质量或花费结果。策略先分类任务，再限制可以参与的角色。

## 分类顺序

分类采用 `first_match`，顺序如下：

1. **大型或高风险**：出现任一高风险边界、新架构或影响范围很广的改动。
2. **机械**：五项 economy 条件必须全部成立：输入和规则确定、操作可独立重试、没有敏感或高风险边界、每项都有自动检查、失败有明确上界。
3. **简单**：文件已知、没有开放判断、可直接验证，且不包含创意或行为变更。
4. **标准**：兜底类别。

## 能力关口

| 类别 | 角色与能力规则 | `strong` 上限 |
| --- | --- | --- |
| 大型或高风险 | 设计审批前 `architect` 必须完成只读决策；`explorer` 负责只读事实收集；批准后由 `implementer` 实现；验证后 `final-reviewer` 必须完成审查 | 2 |
| 机械 | 只有 `batch-worker` 可以执行符合条件的固定规则工作 | 0 |
| 简单 | 主 agent 直接处理 | 0 |
| 标准 | 主 agent 或一个 `implementer` 承担主要实现；`explorer` 与 `reviewer` 按条件使用；两次实质失败后允许一次诊断性的 `architect` 调用 | 1 |

这些数字是策略上限，不承诺调用一定可用，也不表示其身份能够被独立验证。

## 角色边界

- `architect` 和 `final-reviewer` 使用 `strong`，且只读；不得写计划、补丁或代码。
- `implementer` 使用 `balanced`，只能写入已批准范围内的实现、测试和验证。
- `reviewer` 使用 `balanced`，且只读。
- `explorer` 使用 `economy`，且只读；只收集最小事实，不产出设计或编辑。
- `batch-worker` 使用 `economy`，只可在符合条件的固定规则工作中写入，并逐项检查。

主 agent 始终保留分类、审批检查、写入协调、风险决策和最终验证责任。

## 单一编排权

按指令优先级选择一套编排。Model Economy 获得编排权时，是角色拓扑、能力档位、subagent 预算和完成门的唯一事实源。Superpowers 提供 brainstorming、计划、TDD、调试、审查方法和证据验证，但不增加第二套代理工作流。更高优先级指令强制其他工作流时，以该指令为准，Model Economy 只提供成本建议。

标准任务只有在文件位置、依赖或现有事实不明确时才允许 explorer；只有跨模块、关键逻辑、非显然回归风险或实质测试覆盖疑问出现时才允许 reviewer。整个任务最多启动三个 subagent，重新分类不重置预算。

## 审批与验证

创意或行为变更必须在计划或实施前获得已批准设计或可追溯的审批证据。纯文本字面改动，以及完全落在已批准规格内的确定性工作，是例外。

获批后的实现遵循测试驱动开发。声明完成前，必须重新运行与变更相关的验证并检查结果。成本优化不应绕过审批、测试、验证或安全检查。

reviewer 评估变更；final-reviewer 评估大型或高风险任务的残余交付风险；完成前验证核对新鲜证据。三者是不同证据层，不应重复执行完整代码审查。

## 委派前检查

委派前移除密钥、token、个人数据、生产数据、内部 URL 和无关路径，并明确非目标与验证命令。写入前声明每个 agent 的文件集合，集合不得重叠；重叠工作由主 agent 串行协调。整个任务最多启动三个 subagent，同时运行也不得超过三个，禁止递归委派。

## 限制

- 可选的 CodexBar 摘要可以显示本地用量和估算成本，但不建立节省基线，也不进行角色归因。
- 当前 smoke 输出只能观察 Subagent 是否启动；角色和模型身份未验证。
- 适用时，项目指令可以覆盖全局路由。
