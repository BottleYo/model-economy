# 工作原理

Model Economy 是一套工作流策略，不会自动宣称模型质量或花费结果。策略先分类任务，再限制可以参与的角色。

## 安装模式

- **核心模式：**四个 Skill 不依赖本地六角色配置即可运行。简单、机械和标准任务由主 Agent 执行，并使用原生质量门；不声称拥有自定义角色或独立模型身份。
- **增强模式：**六个受管理角色文件、哈希、映射与模板版本全部健康后，才允许使用六角色路由。
- **降级状态：**部分缺失、冲突或过期的增强文件会失败关闭，绝不静默当作核心模式。

核心模式无法满足大型/高风险任务的独立架构师与终审员关口，因此必须报告限制，并让用户选择安装增强模式，或明确批准降低保障的单 Agent 路线。后者不得被报告为完整的 Model Economy 高风险流程。

## 分类顺序

分类采用 `first_match`，顺序如下：

1. **大型或高风险**：出现任一高风险边界、新架构或影响范围很广的改动。
2. **机械**：五项 economy 条件必须全部成立：输入和规则确定、操作可独立重试、没有敏感或高风险边界、每项都有自动检查、失败有明确上界。
3. **简单**：文件已知、没有开放判断、可直接验证，且不包含创意或行为变更。
4. **标准**：兜底类别。

## 增强模式能力关口

| 类别 | 角色与能力规则 | `strong` 上限 |
| --- | --- | --- |
| 大型或高风险 | 设计审批前 `architect` 必须完成只读决策；`explorer` 负责只读事实收集；批准后由 `implementer` 实现；验证后 `final-reviewer` 必须完成审查 | 2 |
| 机械 | 只有 `batch-worker` 可以执行符合条件的固定规则工作 | 0 |
| 简单 | 主 agent 直接处理 | 0 |
| 标准 | 主 agent 或一个 `implementer` 承担主要实现；`explorer` 与 `reviewer` 按条件使用；两次实质失败后允许一次诊断性的 `architect` 调用 | 1 |

这些数字是策略上限，不承诺调用一定可用，也不表示其身份能够被独立验证。

## 角色边界

- `architect` 和 `final-reviewer` 使用 `strong`，且只读；不得写计划、补丁或代码。
- `implementer` 使用 `balanced`，只能写入已批准范围内的实现、测试和验证。
- `reviewer` 使用 `balanced`，且只读。
- `explorer` 使用 `economy`，且只读；只收集最小事实，不产出设计或编辑。
- `batch-worker` 使用 `economy`，只可在符合条件的固定规则工作中写入，并逐项检查。

主 agent 始终保留分类、审批检查、写入协调、风险决策和最终验证责任。

## 原生质量内核

Model Economy 自带四个按条件触发的质量门：意图门只在实质歧义、高风险决策或未批准架构边界存在时阻塞实施；计划门让简单任务直接执行、标准任务使用短计划、大型任务使用持久化计划；测试门按行为风险选择测试先行或替代检查；证据门要求完成前运行并核对新鲜验证。

这些门不是固定流水线。用户目标和验收已经明确时不重复设计审批；文档、纯格式和确定性配置不强制形式化 TDD；验证失败或无法运行时必须如实披露。

## 轻量工程能力

Model Economy 可以在不改变路由的前提下按需加载三个专项叶子 Skill：

| Skill | 触发场景 | 输出 |
| --- | --- | --- |
| `domain-context` | 领域用词、不变量或 ADR 对当前任务有直接影响 | 最小术语表、约束与文档冲突 |
| `module-design` | 需要检查模块边界、耦合或变更面 | 边界问题、接口取舍与最小改进 |
| `disposable-prototype` | 一个具体未知用小实验比继续讨论更便宜 | 隔离实验、观察结果与生产决策输入 |

叶子 Skill 的 `classification_effect` 为 `none`，`subagent_starts` 为 `0`。它们不拥有代理编排权，不创建额外计划或审查，不自行提交。原型不得使用任何真实账户，不得产生任何外部写入或破坏性副作用，也不得从认证文件、环境变量、钥匙串、token cache 或原始会话数据库取得凭据。正式代码仍按原任务类别经过风险匹配的审批、测试和验证。

## 单一编排权与 Superpowers

按指令优先级选择一套编排。Model Economy 默认是角色拓扑、能力档位、subagent 预算和完成门的唯一事实源。安装、启用或发现 Superpowers 不构成完整工作流授权。

只有用户在当前任务中明确要求“完整 Superpowers”或“Superpowers strict mode”时，Model Economy 才把本次编排权交给 Superpowers，并退化为模型与成本建议。单项测试先行、系统化调试或完成前验证请求不等于 strict 授权；交权状态不跨任务保存。

标准任务只有在文件位置、依赖或现有事实不明确时才允许 explorer；只有跨模块、关键逻辑、非显然回归风险或实质测试覆盖疑问出现时才允许 reviewer。整个任务最多启动三个 subagent，重新分类不重置预算。

## 审批与验证

目标存在实质歧义、不同方案会显著改变结果、涉及高风险决策或未批准架构边界时，实施前必须获得可追溯审批。目标与验收明确、可复现缺陷、已批准设计内实施和确定性工作不重复审批。

声明完成前必须重新运行与变更相关的验证并检查结果。成本优化不应绕过必要审批、风险匹配测试、验证或安全检查。

reviewer 评估变更；final-reviewer 评估大型或高风险任务的残余交付风险；完成前验证核对新鲜证据。三者是不同证据层，不应重复执行完整代码审查。

## 委派前检查

委派前移除密钥、token、个人数据、生产数据、内部 URL 和无关路径，并明确非目标与验证命令。写入前声明每个 agent 的文件集合，集合不得重叠；重叠工作由主 agent 串行协调。整个任务最多启动三个 subagent，同时运行也不得超过三个，禁止递归委派。

## 限制

- 可选的 CodexBar 摘要可以显示本地用量和估算成本，但不建立节省基线，也不进行角色归因。
- 当前 smoke 输出只能观察 Subagent 是否启动；角色和模型身份未验证。
- 适用时，项目指令可以覆盖全局路由。
