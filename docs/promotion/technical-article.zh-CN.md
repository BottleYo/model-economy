# 我为什么做 Model Economy：让开发流程随风险伸缩

很多 Agent 开发工作流有一个容易被忽略的问题：流程深度常常由“我们能调用什么”决定，而不是由“当前任务承担什么风险”决定。改一行已知配置、修复跨模块缺陷、重构认证系统，可能被送进同一套长计划、多 Agent 和多轮审查。高风险任务确实需要这些步骤，但把相同成本摊到所有任务上，并不自动带来更好的结果。

Model Economy 的出发点很直接：先分类，再路由。任务按固定顺序判断大型/高风险、机械、简单和标准，首个命中类别决定允许使用的角色、能力档位与质量门。这里的 `strong` 次数和 subagent 数量都是策略级上限，不是平台账单保证，也不证明实际运行了某个模型。

## 两种安装模式

`0.6.0` 把目录安装后的默认状态定义为核心模式。四个 Skill 不依赖用户级六角色文件即可运行：简单、机械和标准任务由主 Agent 处理，同时保留意图、计划、测试和证据四类原生质量门。这样，用户从目录安装 Skill 后不会因为缺少本地角色配置而触发虚假的角色调用。

需要更清晰的角色隔离时，可以通过仓库 CLI 安装增强模式。增强模式包含架构师、实现者、探索者、审查者、批处理者与终审者六个角色。只有角色文件齐全、受管理哈希匹配、模型映射一致、模板版本当前时，它才被视为健康；部分缺失、冲突或过期会进入降级状态并失败关闭。

这个边界在大型/高风险任务上最重要。核心模式无法提供独立架构师与终审员，因此必须明确报告限制，并让用户选择安装增强模式，或批准降低保障的单 Agent 路线。即使批准后者，也不能把结果描述成完整的 Model Economy 高风险流程。

## 和 Superpowers 是什么关系

Model Economy 默认使用自己的轻量质量门，不把“发现或安装 Superpowers”当成启用完整方法论的授权。只有用户在当前任务明确要求“完整 Superpowers”或 strict mode 时，Model Economy 才交出编排权，并只保留模型与成本建议。这个选择不跨任务保存，也不会在同一任务叠加两套 subagent 编排。

## Token 数据怎么处理

项目没有遥测，也不自行扫描原始 session。可选的 `usage` 命令读取用户选择的 CodexBar 白名单摘要，展示总 Token、模型拆分与估算成本，但不会推断角色级 Token。推广期使用公开 A/B 协议记录任务类别、模式、基线/插件工作流、Token、成本估算、subagent、strong 请求、耗时、测试和审查发现；样本不足时不发布固定节省比例。

## 从一个任务开始验证

最简单的体验方式，是用同一个仓库准备三类任务：已知位置的常量修改、可复现的跨模块缺陷，以及涉及认证或权限的架构变更。运行前查看 `status`，再核对实际路线、验证证据与失败关闭行为。任何模型或角色身份如果没有可信事件字段，都应该保持“未验证”。

Model Economy 是社区开源项目，不是 OpenAI 官方产品。项目当前优先收集安装问题、核心模式兼容性和可复算 A/B 样本；网站、支持、隐私、条款和路线图均在仓库公开。欢迎用脱敏、可复现的任务反馈真实效果，也欢迎指出它在哪些场景里增加了不必要的流程。
